# Adopt "Simplicity Over Features" as a Core Rule

No code changes. This plan records a durable decision-making rule that governs every future feature.

## The rule

Before proposing or building anything, it must pass one question:

> **Does this help a business owner make a better marketing decision?**

If the answer is no — or "sort of" — it does not get built. Competitor parity is not a reason. "It would be cool" is not a reason. Saving the user time and sharpening decisions are the only reasons.

## How I will apply it

1. **Every new feature request gets a one-line justification** in my response: what decision it improves, and how much time it saves. If I can't write that line honestly, I'll push back instead of building.
2. **Default answer is "no" or "smaller"** — I will propose the minimum version that delivers the decision-support value, not the complete version.
3. **I will actively recommend removing things** when I notice charts, settings, or screens that don't change what the user does next.
4. **No competitor-driven suggestions.** I won't say "most tools in this space have X" as justification.
5. **Complexity budget applies to UI too** — fewer charts, fewer tabs, fewer toggles, plain-language summaries over dashboards whenever both work.

## Where this gets stored

Saved to project memory as a Core rule so it's applied on every future turn, in every session:

- `mem://index.md` → Core section:
  `Simplicity is the product. Every feature must improve a marketing decision or save time — otherwise don't build it. No competitor-parity features. Prefer removing over adding.`

## What this changes about the v1 MVP already shipped

Nothing to rip out — the shipped v1 (Auth, Onboarding, Dashboard with Health Score + AI Summary + Recommendations, Weekly Reports, Settings) already matches this rule. The deferred v2 list (notifications delivery, PDFs, Slack/Teams, teams, billing, etc.) will each get re-evaluated against the question above before any of them come back on the table — several likely won't.
